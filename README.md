# Mini NPU Simulator

1.  개요

    Python을 이용하여 콘솔 환경에서 실행되는 Mini NPU Simulator를 구현하였습니다.
    이 프로그램은 입력된 2차원 패턴과 필터를 MAC(Multiply-Accumulate) 연산으로 비교하여,
    어떤 패턴이 더 유사한지 판정합니다.

    모드 1에서는 사용자가 3×3 필터 2개와 3×3 패턴을 직접 입력하여 MAC 점수, 판정 결과(A/B/판정 불가), 수행 시간을 확인할 수 있습니다.

    모드 2에서는 data.json 파일에 저장된 5×5, 13×13, 25×25 필터와 패턴을 불러와
    각 케이스의 판정 결과와 expected 값을 비교하고, PASS / FAIL 및 전체 요약 결과를 출력합니다.

6.  핵심 기술 적용

    - MAC 연산 구현

          입력 패턴과 필터의 같은 위치에 있는 값을 각각 곱한 뒤, 그 결과를 모두 더하는 방식으로 MAC 연산을 구현하였습니다.

           ```python
           for i in range(n):
                 for j in range(n):
                    total += pattern[i][j] * filter[i]  [j]
           ```

             1. 입력된 pattern과 filter를 함수 인자로 전달받습니다.
             2. 이중 반복문을 사용하여 같은 위치의 값을 하나씩 꺼냅니다.
             3. 각 위치의 값을 곱한 뒤 total 변수에 계속 누적합니다.
             4. 모든 위치를 순회한 뒤 최종 누적값을 점수로 반환합니다.

           외부 라이브러리를 사용하지 않고 이중 반복문을 이용하여 직접 구현하였으며, 각 위치의 값을 하나씩 계산해 나가는 방식으로 동작하도록 하였습니다. MAC 연산이 곱하고 계속 더하는 과정이라는 것을 직접적으로 이해할 수 있었습니다.

    - 라벨 정규화

      JSON 데이터의 다양한 표현을 통일하기 위해 표준 라벨을 사용하였습니다.

      - '+' → Cross
      - 'x' → X
      - 'cross' → Cross

      라벨 정규화는 expected 값 비교, 필터 이름 처리, 최종 PASS / FAIL 판정 등 여러 위치에서 반복적으로 사용되는 기능입니다.
      만약 이 로직을 각 함수 안에 직접 작성하면 같은 코드가 여러 번 중복될 수 있고, 나중에 새로운 표현이 추가되었을 때 여러 파일을 함께 수정해야 하는 문제가 생깁니다.

      이 문제를 방지하기 위해 normalize_label 함수를 별도로 분리하여 어떤 입력이 들어오더라도 프로그램 내부에서는 Cross 또는 X라는 표준 라벨만 사용하도록 설계하였습니다.

      이렇게 분리하면 코드 재사용성이 높아지고, 라벨 기준이 바뀌더라도 한 곳만 수정하면 되기 때문에 유지보수가 쉬워집니다.
      판정 기준이 일관되게 유지되어 데이터 표현 차이로 인한 오류를 줄일 수 있습니다.

      데이터 표현 차이로 인한 오류를 방지하고, 일관된 기준으로 판정이 가능하도록 구현하였습니다.

    - epsilon 기반 비교

      epsilon는 0에 매우 가까운 작은 값을 의미합니다. 프로그래밍에서 실수 계산을 할때는 내부적으로 오차가 발생할 수 있기 때문에 정확하게 비교하는 것은 어렵습니다.</br>
      부동소수점 연산 과정에서 발생하는 오차를 고려하여 epsilon 기반 비교를 적용하였습니다.

      ```python
      if abs(score_cross - score_x) < 1e-9:
          return "UNDECIDED"
      ```

      두 값의 차이가 epsilon(1e-9)보다 작으면 같은 값으로 판단하도록 하였고 부동소수점 연산에서 발생하는 미세한 오차로 인해 잘못된 판단이 나오는 것을 방지하였습니다.

    - JSON 데이터 처리

      - data.json 파일에서 필터와 패턴 데이터를 로드
      - 패턴 키(size_N)를 분석하여 해당 크기의 필터 선택
      - 필터와 패턴의 크기 일치 여부 검증
      - 크기 불일치 시 프로그램 종료가 아닌 FAIL 처리

      patterns 키에서 크기 N을 추출하는 방식은 data.json의 패턴 키는 size*5_1, size_13_2 와 같은 규칙으로 구성되어 있습니다.
      이 구조를 이용하면 키를 '*' 기준으로 나누었을 때 두 번째 값이 패턴의 크기 N이 됩니다.

      예를 들어 size*13_2 라는 키가 있다면, split('*') 결과는 ['size', '13', '2']가 되고, 여기서 두 번째 값인 13을 추출하여 size_13 필터를 선택하도록 설계하였습니다.

      이렇게 설계한 이유는 패턴과 필터를 별도의 조건문으로 일일이 연결하지 않아도 되고, 새로운 크기의 데이터가 추가되더라도 같은 규칙만 유지하면 자동으로 해당 필터를 찾을 수 있기 때문입니다.

      키 규칙 자체를 이용하여 "패턴 크기 추출 → 같은 크기의 필터 선택 → 크기 검증 후 MAC 연산 수행" 흐름으로 처리되도록 구성하였습니다.

    - 성능 분석

      MAC 연산의 수행 시간을 측정하기 위해 다음과 같이 구현하였습니다.

      - 동일 연산을 10회 반복 수행
      - 평균 시간을 ms 단위로 계산
      - 연산 횟수(N²)와 함께 출력

      출력 항목:

      - 크기 (N×N)
      - 평균 시간(ms)
      - 연산 횟수(N²)

      성능 측정에서는 입력을 받는 시간, print로 출력하는 시간, JSON 파일을 읽는 시간까지 함께 포함하면 순수한 MAC 연산 성능을 정확하게 비교하기 어렵다고 판단하였습니다.

      시간 측정의 시작 시점은 MAC 연산 함수 호출 직전으로 두고, 종료 시점은 함수가 점수를 반환한 직후로 두었습니다.

      사용자의 키보드 입력, 콘솔 출력, 파일 입출력(I/O) 시간은 제외하고 실제 계산이 수행되는 구간만 측정하도록 경계를 잡았습니다. 이렇게 해야 크기별로 증가하는 연산량 자체를 비교할 수 있고, N이 커질수록 시간이 증가하는 이유를 O(N²)와 연결해서 해석하기 쉬워집니다.

9.  결과 리포트

    - 실패 원인
      테스트는 총 3가지 실패 상황을 중심으로 진행하였습니다.</br></br>

      첫 번째는 라벨 불일치입니다.</br>
      패턴 자체는 Cross 형태이지만 expected 값을 X로 설정한 경우입니다.
      이 경우 MAC 연산 결과는 Cross가 더 높게 나오지만, expected 값과 다르기 때문에 FAIL이 발생합니다.
      이를 통해 판정 로직이 올바르게 동작하는지 확인할 수 있었습니다.</br></br>

      두 번째는 크기 불일치입니다.</br>
      패턴의 크기와 필터의 크기가 서로 다른 경우입니다.
      MAC 연산은 같은 크기의 배열에서만 가능하기 때문에, 크기가 다르면 연산 자체가 불가능합니다.
      이 경우 프로그램이 종료되지 않도록 FAIL로 처리하도록 구현하였습니다.</br></br>

      세 번째는 UNDECIDED 상황입니다.</br>
      Cross와 X의 점수가 거의 비슷하게 나오도록 패턴을 구성한 경우입니다.
      두 값의 차이가 매우 작아 epsilon 기준에 의해 UNDECIDED로 판정되며, expected 값과 다를 경우 FAIL이 발생합니다.
      이를 통해 부동소수점 오차와 epsilon 비교의 필요성을 확인할 수 있었습니다.</br></br>

      이 세 가지 테스트를 통해 다양한 상황에서도 프로그램이 안정적으로 동작하는 것을 확인할 수 있었습니다.

    - 시간 복잡도 분석

      MAC 연산은 반복문을 이용하여 모든 값을 하나씩 계산하는 방식으로 구현되어 있습니다.

      ```python
      for i in range(n):
          for j in range(n):
              total += pattern[i][j] * filter[i][j]
      ```

      위 코드를 보면 반복문이 두 번 사용되고 있습니다.
      첫 번째 반복문은 행(row)을 기준으로 돌고, 두 번째 반복문은 열(column)을 기준으로 돕니다.

      즉, 한 번의 계산에서 모든 칸을 하나씩 확인하게 되므로 총 연산 횟수는 "가로 개수 × 세로 개수"가 됩니다.
      이러한 구조를 시간 복잡도로 표현하면 O(N²)이라고 합니다.
      이는 입력 크기가 커질수록 계산량이 빠르게 증가한다는 의미입니다.

      실제 실행 결과에서도 패턴 크기가 커질수록 연산 시간이 증가하는 것을 확인할 수 있었습니다.

10. 생각하기

- 새로운 라벨이 추가된다면 어떤 순서로 확장할 것인가?

  현재 구조에서는 라벨 확장을 위해 크게 3가지 부분을 수정하면 됩니다.

  첫 번째는 judge.py 안에 있는 normalize_label 함수를 수정합니다.
  이 함수는 다양한 입력 값을 프로그램 내부의 표준 라벨로 변환하는 역할을 하기 때문에, 새로운 라벨이 추가되면 해당 함수에 입력 표현을 추가해줍니다.

  두 번째는 data.json에 있는 filters 데이터 구조를 수정합니다.
  각 크기(size_N)에 대해 새로운 필터 데이터를 추가하면, 기존 구조를 그대로 유지하면서도 새로운 라벨을 처리할 수 있습니다.

  세 번째는 judge.py의 judge_scores 함수의 구조를 수정합니다.
  기존의 함수는 Cross와 X 두 개만 비교하는 구조였지만, 라벨이 추가되면 각 라벨의 점수를 딕셔너리 형태로 관리하고 최대값을 기준으로 판정하도록 확장해야 합니다.

  이처럼 입력 정규화 → 데이터 구조 → 판정 로직 순서로 수정하면, 기존 코드를 크게 변경하지 않고도 새로운 라벨을 확장합니다..

- 대규모 패턴 처리 시 어떤 문제가 발생할 수 있는가?

  패턴 크기가 커질수록 가장 먼저 연산량이 크게 증가합니다.
  현재 구조는 모든 위치를 직접 순회하므로 N이 커질수록 계산 횟수는 N²만큼 늘어납니다.
  따라서 25×25보다 더 큰 패턴이나 많은 테스트 케이스를 한 번에 처리하면 전체 수행 시간이 길어집니다.
  예를 들면 1000×1000이라면 1,000,000르 한 번에 처리해야하므로 수행 시간이 25×25보다 배는 더 길어집니다.

  또한 패턴과 필터 데이터를 메모리에 저장하고 반복적으로 접근해야 하므로 데이터가 매우 많아지면 메모리 사용량도 증가하고, 출력해야 하는 로그가 많아질 경우에는 콘솔 출력 자체도 전체 실행 시간을 늘리는 원인이 됩니다.

  그래서 대규모 처리에서는 연산 구간과 I/O 구간을 분리해서 생각해야 하고, 필요하다면 데이터 구조 단순화나 반복 횟수 조정 같은 개선이 필요합니다.

- CPU와 NPU의 병목 차이는 무엇인가?

  CPU는 다양한 작업을 처리할 수 있도록 설계된 범용 프로세서입니다.
  복잡한 제어, 조건 분기, 운영체제 처리에는 강하지만 같은 계산을 매우 많이 반복하는 작업에서는 비효율이 발생할 수 있습니다.

  반면 NPU는 MAC 연산처럼 단순하지만 반복 횟수가 매우 많은 연산을
  병렬로 빠르게 처리하는 데 특화되어 있습니다.
  CPU는 범용성이 강한 대신 대량 반복 연산에서 병목이 생기기 쉽고, NPU는 이런 반복 연산 병목을 줄이기 위해 만들어진 구조라고 볼 수 있습니다.

  이번 과제의 MAC 연산도 작은 크기에서는 CPU로 충분히 처리할 수 있지만, 실제 AI 모델처럼 큰 데이터와 많은 필터를 다루게 되면 CPU만으로는 속도 한계가 생기기 때문에 NPU가 필요한 이유를 이해할 수 있었습니다.
